# REQ-FR-260029: Always-Fresh Backend Query on Every Search Submission

**Status**: Accepted
**Type**: Functional Requirement
**Module**: workspace-search, api-contract
**Affects**: use-graph-data.ts, apollo-client.ts, SearchBar.tsx
**Related ADRs**: ADR-260027, ADR-260023
**Related Requirements**: REQ-FR-260026, REQ-OR-260008

## Summary

Every search submission (Enter key press in SearchBar) must trigger a fresh backend GraphQL query, even if the labels are identical to a prior submission.

## Specification

1. When the user submits a search by pressing Enter in SearchBar:
   - The client immediately invokes a fresh GraphQL query to the backend.
   - The query uses `fetchPolicy: 'network-only'` (or equivalent) to bypass any Apollo Client cache.
   - Cache bypass applies whether the labels match a prior search or are new.

2. The backend receives the complete label list and executes a fresh `retrieve` query.

3. Results are displayed in the graph and result panel, reflecting current database state.

4. This behavior applies to all search submissions, including:
   - First search after workspace entry.
   - Repeated searches with identical labels.
   - Searches after clearing and re-entering labels.

## Rationale

- Ensures frontend state reflects current database truth when external modifications occur (other browser tabs, API clients, Google Sheets integration, etc.).
- Eliminates staleness issues caused by Apollo Client cache reuse for identical queries.
- Provides users with confidence that visible results match the actual backend state.

## Out of Scope

- Caching strategy for non-search queries (e.g., list_labels, schema compatibility).
- Rate-limiting or batching logic for high-frequency repeated searches.
- User-facing indicators for "checking for updates" or "search completed"; let loading state handle this.
