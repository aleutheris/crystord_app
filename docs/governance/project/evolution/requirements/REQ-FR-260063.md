# Requirement: REQ-FR-260063 - Verify-First Account Sign-Up

- **Date**: 2026-06-23
- **Status**: Proposed
- **Type**: Functional Requirement

## Description

The system must let a user create an account through an email-verified, two-step flow: request a code
with `beginSignup(email)`, then create the account with `completeSignup(email, code, password, username)`,
which returns a session token and signs the user in. A `username` is required and a password of at least
12 characters is required (full rules in REQ-CR-260023). Supersedes the sign-up half of REQ-FR-260059
and retires the email-as-username shim (REQ-OR-260013).

## Scenarios

- **Scenario 1: Request a code**
  - **Given** a user enters their email on the sign-up screen,
  - **When** they submit step 1,
  - **Then** the system calls `beginSignup` and advances to a verification-code screen,
  - **And** it shows neutral, non-enumerating copy regardless of whether the email already exists
    (REQ-CR-260024).

- **Scenario 2: Complete sign-up**
  - **Given** a user has received a code,
  - **When** they enter the code plus a valid username and a ≥12-char password and submit,
  - **Then** the system calls `completeSignup`, stores the returned session token, and signs them in.

- **Scenario 3: Wrong or expired code**
  - **Given** a user enters an incorrect or expired code,
  - **When** they submit,
  - **Then** the system shows a form-level error (`SIGNUP-INVALID-OR-EXPIRED-CODE`) and lets them
    re-enter or **resend** the code.

- **Scenario 4: Invalid username or weak password**
  - **When** the server returns `USER-INVALID-USERNAME` or `PASSWORD-TOO-SHORT/-LONG/-COMMON`,
  - **Then** the system shows the error inline on the offending field.

- **Scenario 5: Account created mid-signup**
  - **When** the server returns `SIGNUP-ACCOUNT-ALREADY-EXISTS`,
  - **Then** the system shows a form-level message directing the user to sign in.
