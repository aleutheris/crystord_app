# Requirement: REQ-FR-260069 - Read-Side Access-Level Affordance Gating

- **Date**: 2026-06-23
- **Status**: Implemented (BI-260061, 2026-06-25)
- **Type**: Functional Requirement

## Description

The system must respect the caller's access level on each atom. `AtomOutput` carries `ownerUuid` and
`accessLevel` (`OWNER` / `EDITOR` / `VIEWER`); the client must gate edit/delete/bond-create affordances
accordingly and handle authorization failures. The sharing **write** surface is out of scope (separate
epic); only read-side gating is required here.

## Scenarios

- **Scenario 1: Viewer is read-only**
  - **Given** an atom whose `accessLevel` is `VIEWER`,
  - **Then** the system hides/disables edit, delete, and bond-create affordances for it.

- **Scenario 2: Editor and owner affordances**
  - **Given** `accessLevel` is `EDITOR` or `OWNER`,
  - **Then** the system enables editing and bonding; owner-only actions remain owner-gated.

- **Scenario 3: Authorization failure**
  - **When** a mutation/bond targets an inaccessible atom and returns `AU-UNAUTHORIZED`, or a principal
    is unknown (`CR-16-PRINCIPAL-UNKNOWN`),
  - **Then** the system shows a clear access/principal message via the central error mapper.

- **Scenario 4: Missing access level**
  - **When** `accessLevel` is absent on a response,
  - **Then** the system defaults to the most restrictive (read-only) affordances.
