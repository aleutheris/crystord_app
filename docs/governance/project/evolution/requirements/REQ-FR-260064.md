# Requirement: REQ-FR-260064 - Session Lifecycle and Re-Authentication

- **Date**: 2026-06-23
- **Status**: Proposed
- **Type**: Functional Requirement

## Description

The system must treat the auth token as a revocable, expiring **session** token and recover gracefully
when it ends. A gated operation that fails with `AUTHZ-AUTHENTICATION-REQUIRED` (carried in the GraphQL
error `message`) must be treated as "session expired → route to sign-in", not a hard error. The system
must provide a server-side **Logout** that calls `mutation { logout }`. Supersedes the permanent-token
assumption in REQ-FR-260059.

## Scenarios

- **Scenario 1: Expired session on a gated call**
  - **Given** a signed-in user whose session has expired or been revoked,
  - **When** any gated operation returns `AUTHZ-AUTHENTICATION-REQUIRED`,
  - **Then** the system clears local auth state and routes the user to sign-in (no raw error surfaced).

- **Scenario 2: Server-side logout**
  - **Given** a signed-in user,
  - **When** they choose Logout,
  - **Then** the system calls `mutation { logout }` fire-and-forget and clears local state regardless
    of the network result.

- **Scenario 3: Demo and URL-handoff tokens are sessions**
  - **Given** a demo session or a `?token=` platform handoff session,
  - **When** that session expires,
  - **Then** the same re-authentication behavior applies (no permanence assumption).
