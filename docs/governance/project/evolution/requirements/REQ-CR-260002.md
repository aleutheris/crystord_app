# REQ-CR-260002: Backend Error Handling and Resilience Baseline

- Class: CR
- Status: Accepted
- Date: 2026-04-13
- Source decision: B5 (Error contract)

## Requirement

The frontend shall handle and surface network/timeout errors, GraphQL validation errors, authentication errors, rate limits, and domain-specific errors; deprecation warnings shall be logged but not block operations.

## Rationale

Error resilience is required to prevent cascading failures and to keep users informed of operational issues.

## Scope Notes

- In scope: network timeouts, 4xx/5xx HTTP, GraphQL errors (validation, auth), rate-limit headers, atom-not-found, bond-create-failed errors.
- Out of scope: automatic retry with exponential backoff, error analytics pipelines, user-facing error documentation portal.

## Verification

- Error case tests confirm each error type is caught and handled without crashing.
- Integration tests confirm user sees actionable error messages.
- Regression tests confirm deprecation warnings appear in logs but operations continue.
