## 3. Current Contract

- Contract identifier: backend GraphQL schema, pinned via `backendSchemaRange: "~8.1.0"` (ratified by
  `ICR-260001`). Auto-adopt `8.1.x` patches; `8.2.0`+ requires a new ICR.
- Current behavior/signature: the 8.1.0 schema (auth sessions, atoms/bonds, labels, taxonomy reads,
  startup `schemaInfo` handshake).
- Current consumers: `src/api-contract/*` (operation documents + adapters), the auth, workspace-graph,
  workspace-details, and ui-shell features. The frontend's used surface is **auth + atoms + labels +
  `schemaInfo`** (operations: `BeginSignup`, `CompleteSignup`, `SignIn`, `SignInGoogle`, `Me`,
  `Logout`, password/email/account ops, `CreateAtoms`, `UpdateAtom`, `DestroyAtoms`, `RetrieveAtoms`,
  `ListLabels`, `StartupSchemaInfo`).

## 4. Proposed Change

- Proposed behavior/signature: adopt backend GraphQL schema **9.2.0** as the pinned contract and set
  `backendSchemaRange: "~9.2.0"` (auto-adopt `9.2.x` patches; a higher minor/major requires a new ICR).
- **Net schema delta, 8.1.0 → 9.2.0 (from the bundle diff):** purely **additive**, entirely on the
  taxonomy/`Category*` surface (attributed to backend `BI-260081`, `ADR-260050`):
  - New owner-only queries: `listCategoryDimensionGrants(key): [CategoryGrantOutput!]!`,
    `listCategoryValueGrants(key): [CategoryGrantOutput!]!`.
  - New owner-only mutations: `shareCategoryDimension`, `shareCategoryValue`,
    `revokeCategoryDimensionAccess`, `revokeCategoryValueAccess`,
    `transferCategoryDimensionOwnership`, `transferCategoryValueOwnership`.
  - New non-null fields on existing **output** types `CategoryDimensionOutput` and
    `CategoryValueOutput`: `ownerUsername: String!`, `accessLevel: EffectiveAccessLevel!`
    (additive on a return type — backward-compatible for consumers).
  - New type `CategoryGrantOutput` (mirrors `AtomGrantOutput`).
  - **No removals, renames, signature changes, or newly required inputs** anywhere in the diff.
- Change type: `additive` (net 8.1.0 → 9.2.0). The major-version increment (8 → 9) reflects backend
  taxonomy/access-control evolution that lands **outside** this frontend's used operations; the direct
  8.1.0 → 9.2.0 delta contains no breaking change to any operation the frontend consumes.
- Why a new ICR is required: the `schema-compatibility-contract` rule states a move past `8.2.0`
  requires a new ICR even when additive; `ICR-260001` authorized only `8.1.0`. This ICR ratifies the
  `~9.2.0` pin.

## 5. Impact Analysis

- Technical impact: the `backendSchemaRange` pin only. No frontend code change is required, because
  none of the 9.2.0 additions touch operations the frontend uses (verified: zero Category/taxonomy
  operations in `src/`; the only `taxonomy` reference is an architecture test asserting the pre-auth
  screen does **not** import taxonomy reads).
- Consumer impact: **none** for `crystord_app`. The used surface (auth, atoms, labels, `schemaInfo`)
  is byte-identical between 8.1.0 and 9.2.0.
- Operational impact: re-validate the startup `schemaInfo` handshake against `~9.2.0`; deploy config
  already pins `~9.2.0` across all profiles.
- Risks: minimal. Residual risk is only that a *future* `9.2.x`/`9.3.0` backend change could touch the
  used surface — bounded by the `~9.2.0` range (patch-only auto-adoption) and the fail-fast handshake.
