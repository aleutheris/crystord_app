- Branding source-of-truth lock:
  - `docs/governance/project/branding.md` is authoritative for UI/design-facing behavior.
  - If implementation conflicts with branding policy, branding policy wins unless superseded by explicit ADR decision.
- Mandatory review gate lock:
  - Every UI/design-facing change must include branding-alignment checks in review.
  - Required checks include token intent usage, status semantics, readability/accessibility-safe presentation, and forbidden-pattern avoidance.
- Scope trigger definition lock:
  - UI/design-facing changes include visual styling, token mapping, state/status communication, UX copy tone, and layout/surface changes affecting brand semantics.
- Non-deceptive status communication lock:
  - Status must not rely on color alone; text/icon cues are required.
  - Placeholder and fallback surfaces must be explicit, honest, and non-deceptive.
- Brand token governance lock:
  - UI changes must use approved brand tokens or an explicit token mapping.
  - New token introduction requires governance approval evidence.
- Ownership and exception process lock:
  - Branding governance owner is the repository owner (until formally delegated).
  - Branding exceptions require explicit rationale documented in review notes.
