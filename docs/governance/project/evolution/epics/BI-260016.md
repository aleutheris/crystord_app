## Implementation Decision Locks (Approved)

Approved on: 2026-04-17

- MVP stack baseline lock:
  - React + TypeScript
  - React Router
  - Apollo Client + lightweight local workspace state
  - React Flow (or equivalent editor-grade graph library)
  - Headless UI primitives + utility styling
  - React Hook Form + schema validation
  - Vitest + Testing Library + Playwright
  - Vite + ESLint + strict TypeScript CI checks
- Rejection constraints lock:
  - Reject SSR-first workspace behavior in MVP.
  - Reject stack patterns that erode feature-boundary governance.
  - Reject weak TypeScript/GraphQL ecosystem support.
  - Reject approaches requiring heavy custom glue for core graph workflows.
- Runtime topology lock:
  - Single frontend app with route/layout soft split for public/auth/workspace surfaces.
  - No multi-runtime deployment split in MVP.
- Tooling and CI enforcement lock:
  - Lint, typecheck, unit/integration tests, and critical-flow E2E checks are CI-blocking gates.
  - Dependency/version drift review is required on an explicit cadence.
- Environment separation lock:
  - Demo behavior and production behavior are explicitly separated by environment controls.
  - No production identity-provider integration is required for MVP.
- Auth evolution seam lock:
  - Auth integration remains adapter-based and replaceable without route/layout rewrites.
  - Auth seam preservation is mandatory during MVP changes.
- Verification gates:
  - Stack conformance review confirms allowed baseline usage.
  - Rejection checks confirm no banned stack patterns are introduced.
  - CI evidence confirms required gates pass.
  - Core workflow checks validate selected stack supports graph-first behavior.
  - Backlog Done note includes requirement-linked evidence mapping.
