# Backlog Item: BI-260055 - Verify-First Sign-Up Flow

- **Date**: 2026-06-23
- **Status**: Done
- **Slice**: S1 (unblocker)

## Description

Replace the removed one-shot `signup` with the 8.1.0 two-step, email-verified flow and restore the
ability for new users to register.

## Goals

- Step 1: email → `beginSignup`; advance to a verification-code screen with neutral anti-enumeration
  copy.
- Step 2: code + username + password → `completeSignup`; store the returned session token and sign in.
- Add a **Resend code** action (~45s cooldown) and a code-expiry hint.
- Add a single shared username/password validation schema reused here, in `setPassword`, and in reset.
- Map sign-up errors via the central mapper: inline for `USER-INVALID-USERNAME` / `PASSWORD-*`,
  form-level for `SIGNUP-INVALID-OR-EXPIRED-CODE` / `SIGNUP-ACCOUNT-ALREADY-EXISTS`.

## Acceptance Criteria

- A new user completes email → code → username+password and lands signed-in.
- Wrong/expired code shows a form-level error and allows re-enter/resend.
- Invalid username / weak password show inline field errors; rules match REQ-CR-260023.
- `beginSignup` never reveals account existence (REQ-CR-260024).
- **Branding & UX:** the verification-code screen and both sign-up steps comply with
  `docs/governance/project/branding.md` (Trust Blue primary actions, Control Green only for ownership/confirm,
  neutral surfaces, brand tokens via `@ui`/CSS custom properties — no hardcoded hex per REQ-CR-260011),
  and extend the existing unified `SignInPage` two-column auth + demo layout (REQ-CR-260020 /
  ADR-260053) rather than a bespoke screen.
- **Accessibility:** code-entry and fields are keyboard-navigable with labelled inputs, visible focus,
  `role="alert"` errors, and `aria-live` on the resend/cooldown status (non-color-only state cues).
- **Clean code:** files ≤ 200 lines, functions ≤ 30 lines, no dead code / unused imports, meaningful
  naming; `use-*` hook owns flow/behavior, components own presentation (per project slicing practice).
