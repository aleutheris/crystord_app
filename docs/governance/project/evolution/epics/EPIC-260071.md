# Epic: EPIC-260071 — Table View: spreadsheet-style peer to the graph

- Epic ID: EPIC-260071
- Title: Table View — spreadsheet-style center view
- Persona/Actor: Excel/Sheets-minded users; anyone scanning/editing many atoms at once.
- Value statement: For spreadsheet-minded users, deliver a familiar table/grid view so atoms can be scanned, sorted, and edited like rows — the most familiar on-ramp.
- Context: The user guide pitches Crystord as "one spreadsheet row = one Atom" and targets Excel/Sheets users, yet there is no spreadsheet-like view. The demoted left flat-list (`SearchResultPanel`) finds its proper home here.
- Problem statement: No tabular representation of the working set; the flat atom list was orphaned in the left rail (a representation living where navigation belongs).
- Expected outcome: A **Table** center view — columns: title · labels · category facets · content · type · status — sorted by `updatedAt` (backend default), inline-editable cells. Registered as a center view; shares the selection model with the graph views.
- Success metric(s): Users view, sort, and inline-edit atoms in a grid; selection is shared with Flow/Network; Table is a first-class peer.
- Non-goals: Not "home" (the node-graph is, per Q1); the kanban board (EPIC-260075).
- Constraints: Q1 — Table is a strong first-class peer that supports the graph, not a competitor for "home."
- Dependencies: EPIC-260066 (view registry, shared selection/working-set contracts).
- Risks/assumptions: Inline-edit cells reuse the same mutation path as the inspector.
- Target window: TBD (Ideation).
- Validation approach: Unit tests for column rendering/sort + inline edit; E2E editing a cell and seeing it persist + reflected in the graph.

- [ ] Inline-edit cell behavior and its mutation path.
