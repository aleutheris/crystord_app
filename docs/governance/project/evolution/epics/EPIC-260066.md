# Epic: EPIC-260066 — Foundation: App Shell, Slot Registries & Shared Widgets

- Epic ID: EPIC-260066
- Title: Foundation — App Shell, Slot Registries & Shared Widgets
- Persona/Actor: All Crystord users (indirectly); the team building every later UI capability.
- Value statement: For the product team, deliver a stable four-region UI shell with extension-point registries and a shared widget kit so that every later capability ships as an independent, non-blocking leaf.
- Context: The deployed backend (schema 9.2.0, pinned `backendSchemaRange: "~9.2.0"`) is far ahead of the UI, which still exposes only the old atoms+labels+bonds model — roughly 30% of the product. Exposing the rest (categories, compute, collaboration, account) needs a structural spine, not more ad-hoc panels.
- Problem statement: Today's `WorkspaceShell` wires features ad hoc — a conditional left result list, a single untabbed `DetailPanel`, two hardcoded canvas tabs. Without registries + shared-state contracts + shared widgets, each new feature would restructure the shell and collide with others, so independent delivery is impossible.
- Expected outcome: A shell where Top bar / Left rail / Center view-host / Right inspector each have one job and expose a registry (navigators · views · inspector tabs); shared-state contracts (working set, selection, preferences-with-defaults) owned by the shell; a shared widget kit (filter builder, permission-aware category tree, label chip editor); and a node-extension seam on `AtomNode`. Ships working on today's content (a trivial Labels navigator, the existing Flow/Network views, a basic Details inspector tab) with no new feature logic beyond the shared widgets.
- Success metric(s): Every later leaf (EPIC-260067…260075) adds a navigator/view/inspector-tab by registration only, importing no other feature (enforced by `src/architecture.test.ts`); the shell refactor keeps all existing E2E green.
- Non-goals: No new end-user feature behavior (categories UI, compute UI, account dropdown, etc. are leaves); no charts/analytics libraries (REQ-CR-260010); no real-time-collaboration engine (REQ-CR-260010); no SSR / module-federation.
- Constraints: Must extend, not fight, the enforced architecture — cross-feature import ban, `ui-shell`→barrel-only imports, adapter boundary (REQ-FR-260021), dual-view shared selection (ADR-260032 / REQ-FR-260034); branding via `branding.md` (REQ-CR-260011); CSR-first single-app.
- Dependencies: None — this is the root; all other epics depend on it.
- Risks/assumptions: Refactoring the shell risks regressing existing graph/search/detail behavior — mitigate by treating the existing E2E suite as the contract. Assumes the three registries + shared-state contracts are the right decomposition (already matches the enforced architecture).
- Target window: TBD (Ideation).
- Validation approach: Existing E2E (sign-in, graph-workspace, search, create-atom) stay green; new architecture tests assert the registries exist and that features register through them with no cross-feature imports.

## Product context this epic carries (self-contained — the source brainstorm file is retired)

**Headline.** Backend at schema 9.2.0; UI shows ~30%. The written user guide (`docs/user-guide.md`) is current-ish (schema 8.1.0, documents categories + compute) — the laggard is the **UI**, not the docs.

**The five verbs** (each needs a home): **Discover** (`list_labels`/`listLabels`, `retrieveCategoryBrowse`, `retrieve`), **Classify** (labels + hierarchical categories: dimensions→values), **Relate** (bonds — the graph, done), **Compute** (`operation`: SUM/MINUS/PRODUCT/DIVIDE/COLLECT + dependency graph), **Collaborate/Audit** (workspaces, sharing/grants, change history).

**Substrate vision (standing direction).** The graph + extensible compute engine is a shared core; other apps (documentation, notes) are *configurations* of it, each needing its own concrete front door rather than "graph+flow" as a universal UI — the Blender lesson: the node graph is the engine, the viewport is home. Operations are open/extensible (`discoverOperations`, string-named `operation`, JSON `content`), so e.g. a "document" could later be a CONCAT/compose operation with no contract change. **Analyst is v1; the hybrid is the destination.**

**The stable shell (what this epic builds).** Four regions, each one job, each a registry slot:
- **Top bar** — context + query/facet chips · account menu (query-state; settings routes). Today: `<header>` (brand · `SearchBar` · `ThemeToggle` · Account · Sign Out) + `QuerySummary`.
- **Left rail** (collapsible) — navigate the *index*; registry of **navigators** (Labels, Categories, …). Today: `SearchResultPanel` (flat list — to be demoted).
- **Center** (view-host) — represent the *working set*; registry of **views**: **Flow** (dependencies/compute) · **Network** (relationships) · **Table** · **Board**. The node-graph (Flow/Network) is home. Today: `GraphViewTabs` switching `GraphCanvas` (= Flow view) / `NetworkCanvas`.
- **Right rail** (collapsible) — inspect/edit the *selection*; registry of **inspector tabs**: Details · Classify · Compute · History · Share. Today: `DetailPanel` (single, untabbed).

**Decoupling glue — three shared-state contracts, shell-owned:** working set (filters → atoms in scope; extends `useSearch`), selection (`selectedAtomId`, already lifted, fed to every view), preferences-with-defaults (extends `feature-flags.ts`; home of the Q3 emphasis switch's default). No feature imports another feature.

**Shared widget kit (built once here):** the filter builder (labels + category facets — search + category browser + COLLECT), the permission-aware category tree (left-rail navigator + atom "pick" mode), the label chip editor. The atom-reference picker is single-consumer and lives in EPIC-260069. The **node-extension seam** on `AtomNode` lets compute badges (EPIC-260069) and LOD/Blender blocks (EPIC-260072) layer without colliding.

**Dependency shape:** this Foundation is the root; feature epics are mutually independent leaves — each touches slots and shared contracts, never each other.

**Standing decisions (resolved).**
- **Q1** — Primary user is the graph/knowledge person; the node-graph (Flow/Network layouts) is home; Table is a strong first-class peer, not home.
- **Q2** — Everybody authors taxonomy inline (no admin-only surface); the category tree carries edit affordances everywhere; taxonomy will later gain ownership/permissions mirroring atom sharing (planned, not built) — build the tree against that permission model from the start.
- **Q3** — Computed atoms are a core daily feature; compute-as-differentiator (analyst v1), realized with the current stack (Flow, status badges, "Explain this value", Table) — no charting library. Keep emphasis (Network-vs-Flow prominence, always-on vs on-demand badges) a configurable switch whose UI home is the L6 Preferences pane.

**Standing constraints (from the enforced architecture).**
- Charts/analytics libs (d3, chart.js, recharts, nivo, plotly, victory) are excluded by REQ-CR-260010; adding charts later requires a deliberate ICR. Compute (EPIC-260069) ships without them.
- Sharing is **async** (grants/roles + change history), not live multiplayer — RTC engines excluded by REQ-CR-260010.
- Branding: desaturated brand palette, never color alone (REQ-CR-260011 / `branding.md`).
