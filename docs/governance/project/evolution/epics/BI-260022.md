- Backlog Item ID: BI-260022
- Title: Add temporary capped recommendation strategy from list_labels
- Persona/Actor: Data manager and domain operator
- Value statement: For workspace users, constrain pre-search recommendations to a small useful set so discovery remains helpful without overwhelming the UI at scale.
- Context: Recommendation visibility is accepted, but unbounded label counts create scalability and usability risk.
- Problem statement: Showing all labels as recommendations does not scale; users need a concise, practical set until a per-user most-used strategy exists.
- Expected outcome: Before first search, users see exactly three recommended labels selected from `list_labels`; future per-user ranking is deferred.
- Success metric(s):
  - Temporary source and cap policy are encoded in requirement and backlog traceability.
  - Recommendation panel remains concise even when total labels are large.
- Acceptance criteria (behavior):
  - REQ-OR-260007 and REQ-CR-260012 are created and linked.
  - Product backlog and backlog status include BI-260022 with traceability.
  - Instruction todo items for source and scale decisions are marked done.
- Non-goals: Implementing per-user most-used labels query and persistence in this slice.
- Constraints: Keep solution temporary and compatible with a future backend query for most-used labels.
- Dependencies: REQ-FR-260023, REQ-FR-260022, existing `list_labels` backend query.
- Risks/assumptions:
  - Risk: random selection may vary between sessions and feel inconsistent.
  - Assumption: temporary variability is acceptable until personalized ranking is implemented.
- Validation approach: Governance traceability checks now; implementation behavior checks in follow-on slice.
- Implementation scope (when item moves to In Progress):
  - In-scope modules/files: `workspace-search`, `api-contract`, relevant tests.
  - Out-of-scope guardrails: new backend query design and user-preference persistence.

  - Regression checks: blank-first entry and recommendation visibility preserved.
  - Risk and rollback notes: if random recommendations are rejected, fallback is to hide recommendations until source/ranking redesign.
