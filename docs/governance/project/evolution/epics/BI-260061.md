## Description

Gate edit/delete/bond affordances on the caller's `AtomOutput.accessLevel` (`OWNER`/`EDITOR`/`VIEWER`)
so the UI never offers actions the backend will reject, and handle the related authorization errors.

## Goals

- Thread `accessLevel` (and `ownerUuid`) from `retrieve` results into `workspace-graph` and
  `workspace-details`.
- `VIEWER` → read-only affordances; `EDITOR` → edit/bond; `OWNER` → full. Missing level → most
  restrictive.
- Handle `AU-UNAUTHORIZED` and `CR-16-PRINCIPAL-UNKNOWN` via the central error mapper.
- Harden expiry handling (noted in the BI-260056 review): gated-read hooks (e.g. `use-graph-data`)
  should recognize a reauth code (`AUTHZ-AUTHENTICATION-REQUIRED`) and skip setting a raw error, so the
  "no raw error on session expiry" guarantee no longer relies on the shell unmounting before the
  hook's error renders.

## Acceptance Criteria

- A `VIEWER` atom shows no edit/delete/bond affordances; `EDITOR`/`OWNER` show the appropriate set.
- Attempting a mutation on an inaccessible atom surfaces a clear access message, not a raw error.
- Responses lacking `accessLevel` default to read-only.

## Non-goals

- Share/transfer write UI and workspaces (separate sharing epic).

## Done Evidence Block

- **Gating module — `api-contract/access-control.ts` (new, 100% line/branch/func).**
  `atomPermissions(level) → { canEdit, canDelete, canBond }` encodes the backend rules
  (`docs/user-guide.md` §Access: `change` needs Editor-or-higher; `destroy` is owner-only): OWNER → all;
  EDITOR → edit/bond, no delete; VIEWER / missing → read-only (most restrictive, REQ-FR-260069 §4).
  Exported from the api-contract barrel; both features consume it (no cross-feature import).
- **Detail panel (`workspace-details/DetailPanel.tsx`, 100%).** For an existing atom it gates on the
  caller's level: VIEWER → inputs `readOnly`, no Save, no Delete, a "view-only access" note; EDITOR →
  editable + Save, no Delete; OWNER → Save + Delete. Creation stays fully editable. `handleSave` also
  guards on `canEdit` so an Enter-submit on a read-only atom dispatches nothing.
- **Canvas affordances (`workspace-graph`).** Bond-create handles are hidden on non-bondable atoms:
  `graph-types.atomsToNodes` + the network node builder thread `canBond`, and `AtomNode` /
  `CircleAtomNode` drop their source (Right handle / hover ring) when `canBond === false`. The shared
  `use-canvas-interactions` is the behavioral chokepoint for both views — it blocks bond-create from a
  non-bondable source, atom-delete on a non-owned atom, and bond-removal on a non-editable source.
- **Failure handling (`use-graph-data`, AC2 + expiry hardening).** Retrieve/mutation failures are
  interpreted by the central mapper: a known access/principal failure surfaces a clear message (not a
  raw string); a session-expiry code is left to the global sign-out (the hook no longer flashes a raw
  error in its place — closes the BI-260056 review note); unknown mutation errors keep the prior silent
  behavior, unknown retrieve errors keep their raw message.
- **Tests.** New: `access-control.test.ts`, `AtomNode.test.tsx`; extended canvas-interactions,
  graph-types, CircleAtomNode, DetailPanel, and use-graph-data tests. E2E `access-gating.spec.ts` (+3 →
  47): VIEWER read-only, EDITOR edits-not-deletes, OWNER full, driven through search → result → detail.
- **Coverage.** Gating module `access-control.ts` 100% (the BI bar); DetailPanel / AtomNode /
  graph-types 100%. `use-canvas-interactions`, `use-graph-data`, and `CircleAtomNode` retain
  pre-existing gaps in paths untouched by this slice (bond-confirm/undo handlers, `createAtom`, hover
  timers); the gating/error branches added here are covered by the new tests.
