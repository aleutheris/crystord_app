# BI-260028: Guarantee Fresh Backend Fetch on Every Search Submission

**Status**: Done
**Type**: Feature
**Module**: workspace-search, api-contract
**Component**: src/features/workspace-graph/use-graph-data.ts, src/api-contract/apollo-client.ts, src/features/workspace-search/SearchBar.tsx
**Blocks**: Search freshness and database consistency

## Dependencies

- REQ-FR-260029: Always-Fresh Backend Query on Every Search Submission
- ADR-260027: Always-Fresh Search Fetching and Auto-Chip on Enter

## Acceptance Criteria

1. Every search submission (Enter key press) triggers a fresh backend GraphQL query.

2. Apollo Client cache is bypassed for search queries:
   - Use `fetchPolicy: 'network-only'` in `fetchAtoms()` when called from `search()`.
   - OR configure Apollo cache to not cache `retrieve` field results.

3. Fresh fetch applies regardless of label repetition:
   - Search with [label1, label2].
   - Search again with [label1, label2].
   - Both trigger backend queries; second is not skipped.

4. Results reflect current database state:
   - If atoms were deleted via other means (API, other tabs) between searches, second search reflects deletions.

5. All existing tests pass; new tests cover fresh-fetch behavior.

## Implementation Notes

- Update `use-graph-data.ts` `search()` method to ensure `fetchAtoms()` uses `fetchPolicy: 'network-only'`.
- Alternatively, enhance Apollo cache configuration in `apollo-client.ts` to explicitly bypass caching for `retrieve` field.
- Consider performance implications for high-frequency repeated searches (backend load, network bandwidth).

## Verification

- Unit test: repeated identical search triggers two backend queries (not cached).
- Unit test: results from second search reflect any database changes made between searches.
- E2E test: search in one tab, modify database in another tab, search again in first tab, verify updated results.
- Check Apollo DevTools to confirm no cache hits for `retrieve` queries.
