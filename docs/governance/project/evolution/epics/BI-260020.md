- Backlog Item ID: BI-260020
- Title: Rebaseline workspace entry to blank-first explicit search
- Persona/Actor: Data manager and domain operator
- Value statement: For workspace users, deliver a blank-first entry model so the first visible graph state is driven by explicit search intent rather than implicit preload behavior.
- Context: Post-MVP UX correction after observing that preloaded atoms imply an automatic query and reduce trust in the search-driven workspace model.
- Problem statement: The current workspace preloads graph data on entry, which conflicts with the desired blank-first experience and makes it unclear whether visible atoms reflect an intentional user query.
- Expected outcome: Users sign in to an empty workspace shell, perform an explicit search, and only then see matching atoms rendered in the graph.
- Success metric(s):
  - Blank-first entry behavior is encoded in traceable requirements.
  - Backlog scope is clear enough to implement without reopening the entry-model decision.
- Acceptance criteria (behavior):
  - REQ-FR-260022 is created and linked.
  - REQ-FR-260007 is deprecated rather than deleted.
  - REQ-FR-260019 reflects blank workspace entry followed by explicit search graph load.
  - Product backlog and backlog status include BI-260020 with traceability.
- Non-goals: Detailed search-submit UX, suggestion ranking, autocomplete behavior, and backend search freshness policy.
- Constraints: Keep existing folder structure and governance model; preserve sign-in flow and workspace shell routing.
- Dependencies: Existing auth-entry flow, workspace shell behavior, and subsequent search-model decisions.
- Risks/assumptions:
  - Risk: a blank workspace may feel empty without clear search guidance.
  - Risk: follow-on search-model decisions may still require additional requirement changes.
  - Assumption: explicit search intent is preferred over default preload behavior.
- Validation approach: Requirement/index traceability checks now; implementation validation later via end-to-end workspace-entry checks.
- Implementation scope (when item moves to In Progress):
  - In-scope modules/files: `ui-shell`, `workspace-graph`, `workspace-search`, related E2E coverage.
  - Out-of-scope guardrails: autocomplete, ranking logic, and broader search redesign beyond first-load behavior.
- Evidence requirements for Done:
  - Requirement-to-evidence mapping: REQ-FR-260022 -> workspace entry E2E/manual evidence; REQ-FR-260019 -> critical flow update evidence.
  - Quality gates: `npm run lint`, `npm run typecheck`, `npm run test`, `npm run test:e2e`.
  - Regression checks: demo sign-in entry and explicit search graph load.
  - Risk and rollback notes: preserve route/auth stability and provide quick rollback to previous load behavior if needed.
