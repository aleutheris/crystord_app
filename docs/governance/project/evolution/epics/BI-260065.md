# Backlog Item: BI-260065 - Promote Shared Auth UI Primitives to ui-primitives

- **Date**: 2026-06-25
- **Status**: Done
- **Slice**: Maintainability / DRY (surfaced during the BI-260059 review)
- **Ready since**: 2026-06-25 ‚Äî pure refactor; `src/ui-primitives/` scaffold already exists (ADR-260051).
- **Done**: 2026-06-25 ‚Äî clears the last item of the Auth & Authz epic backlog.

## Description

The feature-boundary rule forbids one feature module from importing another's internals. As a result,
auth UI is duplicated across `auth-entry` and `account-settings`:
- the accessible labelled field (`auth-entry/AuthField`) is re-implemented as a plain `<input>` in
  `account-settings/AuthMethodsSection`;
- the GIS credential dance (`auth-entry/GoogleSignInButton`) is re-implemented as
  `account-settings/GoogleLinkButton`.

Both belong in the shared `ui-primitives/` template library (ADR-260051), which both features may
import without crossing a feature boundary.

## Goals

- Move the accessible field to a shared `ui-primitives` form primitive; consume it from sign-up,
  password-reset, and the account-settings password form.
- Extract a shared GIS credential button (yields an ID token via `onCredential`) into `ui-primitives`;
  `GoogleSignInButton` (sign-in ‚Üí `signinGoogle`) and the account-settings link flow (‚Üí `linkGoogle`)
  both consume it. Keep the latest-ref guard so the GIS button is not re-initialized on parent
  re-renders.
- Remove the duplicated `AuthField` copy and the duplicated GIS dance.

## Acceptance Criteria

- One shared field primitive and one shared GIS credential primitive in `ui-primitives`; no duplicated
  copies remain in features.
- No behavior change to sign-up, sign-in, reset, or account-settings flows.

## Non-goals

- New auth behavior or actions.

## Dependencies

- None. Pure refactor over BI-260048/055/057/059 code.
