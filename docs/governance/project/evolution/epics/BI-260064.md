# Backlog Item: BI-260064 - Shared Modal Focus Management for Overlay Dialogs

- **Date**: 2026-06-25
- **Status**: Done
- **Slice**: Accessibility polish (surfaced during the BI-260058 review)
- **Ready since**: 2026-06-25 — independent a11y polish over existing modals; no dependencies/unknowns.
- **Done**: 2026-06-25

## Description

The overlay dialogs declare `aria-modal="true"` (which tells assistive tech the background is inert)
but implement no focus management — no focus-on-open, no focus trap, and no focus restore on close. So
keyboard focus can still tab to the content behind the dialog, contradicting the `aria-modal` promise.
This affects the new **account-settings** modal (BI-260058) and the pre-existing **atom-creation**
modal (BI-260046); both should be fixed by one shared utility rather than ad hoc per modal.

## Goals

- A shared focus-management primitive (hook or wrapper component) that, for an open dialog:
  - moves focus into the dialog on open (e.g. first focusable / the close control),
  - traps Tab/Shift+Tab within the dialog while open,
  - restores focus to the triggering control on close.
- Apply it to `AccountSettingsPanel` (account-settings) and the atom-creation `DetailPanel` overlay in
  `WorkspaceShell`.
- Scope the modal's Escape handling to the dialog so it no longer also reaches the canvas Escape
  handler underneath (BI-260058 review finding #2 — naturally resolved by a proper trap).

## Acceptance Criteria

- With a dialog open, Tab/Shift+Tab cycle only within the dialog; focus is on a dialog control on open
  and returns to the trigger on close.
- Escape closes the dialog without also mutating the canvas behind it.
- No change to the dialogs' visible behavior or content.

## Non-goals

- New dialog content or actions; this is focus/keyboard behavior only.

## Dependencies

- None. Independent a11y polish over existing modals.

## Done Evidence Block

- **Shared primitive — `src/a11y/use-modal-focus.ts` (new, 100% line/branch/func).**
  `useModalFocus(containerRef, onClose)`: on mount focuses the first focusable in the dialog (else the
  container), traps Tab/Shift+Tab (wrapping at the ends), routes Escape to `onClose` with
  `stopPropagation` (so it can't reach handlers behind the dialog), and restores focus to the trigger on
  unmount. Lives outside the feature tree (`src/a11y/`) so both a feature and ui-shell can consume it
  without a cross-feature import. Latest-ref on `onClose` keeps a changing callback from tearing down the
  trap.
- **Account-settings modal.** `AccountSettingsPanel` now uses the hook (dialog `ref` + `tabIndex={-1}`)
  and the old window-level Escape listener was removed — Escape is now dialog-scoped, resolving the
  BI-260058 review finding #2 (Escape no longer also reaches the canvas).
- **Atom-creation modal.** Extracted the inline overlay from `WorkspaceShell` into
  `ui-shell/AtomCreationOverlay.tsx`, which applies the same hook — so the creation panel now gets
  focus-on-open, a Tab trap, scoped Escape-to-close, and focus restore (none of which it had before).
- **Tests.** `use-modal-focus.test.tsx` (focus-on-open, container fallback, Escape + stopPropagation,
  Tab/Shift+Tab wrap, middle-Tab no-op, no-focusable trap, ignore-other-keys, restore-on-close,
  latest-onClose, null-container guard). `AccountSettingsPanel` retained its Escape/close tests
  unchanged (now satisfied by the scoped handler); `WorkspaceShell` creation-flow tests cover
  `AtomCreationOverlay`.
- **Coverage.** `use-modal-focus.ts`, `AtomCreationOverlay.tsx`, and `AccountSettingsPanel.tsx` all 100%.
- **Commands.** `npm run lint` → 0; `npm run typecheck` → 0; `npm run test` → 0 (915 passing, 54 files);
  `npm run test:e2e` → 47 passed.
- **Rollback.** Revert the listed files; modals lose the focus trap (and the account modal returns to the
  window-level Escape listener).

### Post-review polish (2026-06-25)

- **Fixed (review finding) — creation overlay was a trap without dialog semantics.** `AtomCreationOverlay`
  had the focus trap but no `role="dialog"`/`aria-modal`, so a screen reader got the trap without being
  told it was a modal — the inverse of the BI-260058 finding #1, and inconsistent with
  `AccountSettingsPanel`. Since this is new code from this slice, completed it here: added
  `role="dialog" aria-modal="true" aria-label="Create atom"` to the overlay container (the trap it
  already has is exactly what `aria-modal` requires to be honest). Added a `WorkspaceShell` test asserting
  the creation overlay is an accessible modal dialog. Re-verified: lint/typecheck 0, 916 unit tests, 47
  e2e, new modules still 100%.
