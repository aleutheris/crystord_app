# ADR-260056: Session-Token Lifecycle, Re-Authentication, and Server-Side Logout

- **Date**: 2026-06-23
- **Status**: Accepted

## Context

In 8.1.0 every `signin` / `completeSignup` / `signinGoogle` mints a **session** token that expires
(~24h idle / ~30d absolute) and can be revoked (`logout`, `revokeAllSessions`, password reset, email
change). The current client treats the token as permanent, persists it to `localStorage`, and only
auto-signs-out on HTTP `401` or `extensions.code === 'UNAUTHENTICATED'`. Because 8.1.0 returns codes in
the GraphQL **`message`** (`AUTHZ-AUTHENTICATION-REQUIRED`), the existing detector will **not fire** —
a silent breakage. This supersedes the permanent-token assumption of ADR-260048/ADR-260050.

## Decision

1. **Expiry detection rewrite (D-07):** detect auth failure by parsing the GraphQL error `message` for
   `AUTHZ-AUTHENTICATION-REQUIRED` (via the central mapper, ADR-260057) and trigger the existing
   session-expired channel → route to sign-in. Keep the HTTP-`401` path as a fallback.
2. **Persistence & boot policy (D-08):** keep `localStorage` persistence; perform **lazy** re-auth on
   the first failed gated call rather than a blocking `me` probe at startup (avoids added startup
   latency). An optional `me` probe may be added later behind no UX regression.
3. **Server-side logout (D-09):** `signOut()` calls `mutation { logout }` **fire-and-forget**, then
   clears local state regardless of the network result. The Logout control lives in the shell header /
   account menu.
4. **Sign out everywhere (D-10):** `revokeAllSessions` is deferred to S2 (account-management surface,
   ADR-260058).
5. **Demo as session (D-18):** the demo account (owner-provisioned, invalid-email + fixed password) is
   treated as an ordinary session — same lazy re-auth, no special casing; the demo token stays
   non-persisted per ADR-260050.
6. **URL token is a session (D-19):** the platform token accepted from `?token=` (ADR-260052) is a
   session token subject to the same expiry/revocation; remove any permanence assumption downstream.

## Alternatives Considered

- **Silent token refresh / blocking boot validation:** rejected for S1 — adds startup latency and
  complexity; lazy re-auth covers the failure path adequately.
- **Awaiting `logout` before clearing local state:** rejected — a slow/failed network call must not
  trap the user in a signed-in UI.

## Consequences

- **Positive**: expired sessions route to re-auth instead of surfacing raw errors; real server-side
  session termination on logout.
- **Negative**: auth-state code grows; demo tabs can expire mid-use (acceptable).
- **Risks**: missed error-message shapes; mitigated by centralizing detection in the mapper.

## Verification

- Unit: message-based expiry detection, logout fire-and-forget, demo non-persistence, URL-token
  session handling. E2E: expire → redirected to sign-in; logout ends session. 100% coverage of changed
  auth-state modules.

## Rollback

- Revert to the 401-only detector and local-only logout (re-exposes the silent-expiry gap).
