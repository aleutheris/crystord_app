# ADR-260039: Flow View Directional Projection, Relayout Contract, and Visual Grammar Baseline

## Workflow

- This ADR records accepted Flow-view behavior decisions for directional projection, relayout controls, and visual grammar.
- It formalizes how Flow view differs from Network view while preserving shared source-dataset behavior.
- It enables implementation scope for BI-260044 and linked requirements.

Status: Accepted
Date: 2026-05-19
Owners: Repository owner, product collaborator
Superseded by: ADR-260040 (D1 only)

## Context

Flow view currently shares the same fetched dataset with Network view but lacks a strict flow-specific rendering contract. Product direction requires Flow view to behave as a readability-first directional flow diagram, not as a generic full-graph rendering.

Decisions were explicitly made to:

1. Make Flow view left-to-right only in MVP.
2. Filter Flow view to eligible bonds and participating nodes only.
3. Keep an MVP-simple eligibility seam in frontend configuration.
4. Add predictable relayout controls with selection/viewport continuity.
5. Align Flow edge/handle grammar with horizontal directional flow.

## Decision

### D1: Flow View Projection Semantics
- Flow view renders a deterministic projection over the shared fetched dataset.
- Projection includes only bonds from an eligible-bond allowlist and nodes participating in at least one eligible bond.
- Initial eligible-bond allowlist baseline is OP_DEPENDENCY.
- Eligibility configuration is frontend-managed in MVP with an extension seam for future additions.

### D2: Orientation and Scope Boundary
- Flow-view relayout orientation is left-to-right only in MVP.
- Top-to-bottom orientation toggle is explicitly deferred as future scope.

### D3: Relayout Contract
- Flow view exposes a visible relayout control.
- Relayout is available whenever Flow view is active in this batch.
- Relayout preserves selected atom and viewport focus.
- Manual node repositioning remains allowed after relayout.

### D4: Layout Quality Priorities
- Flow layout quality priorities are fixed as:
  1. Minimize edge crossings.
  2. Preserve hierarchical layering.
  3. Apply symmetry as best-effort.
- Cyclic or non-hierarchical inputs remain left-to-right; unavoidable back-edges/crossings are acceptable.
- Layout algorithm identity is an implementation detail governed by acceptance criteria and tests.

### D5: Flow Visual Grammar
- Directed Flow edges render with visible arrowheads.
- Bond labels are not rendered in Flow view for this batch; label data seam is retained.
- Flow nodes use universal left/right handles in MVP.
- Existing node shape remains unchanged in this batch.

## Alternatives Considered

- Option A: Keep Flow view as an unfiltered full-graph rendering. Rejected: poor readability for flow-specific interpretation.
- Option B: Support both left-to-right and top-to-bottom immediately. Rejected: adds UX and test complexity before baseline stabilizes.
- Option C: Backend-driven eligible-bond configuration in MVP. Rejected: unnecessary complexity for current scope.

## Consequences

### Positive
- Flow view becomes purpose-specific and easier to read.
- Mixed search results can still be consumed via Network view while Flow remains semantically constrained.
- Relayout becomes predictable without disrupting user context.

### Negative
- Flow view may hide part of search results by design, requiring clear user messaging.
- Additional projection and layout tests are required to prevent regression.

### Risks
- If eligibility allowlist grows without governance, Flow semantics may drift.
- Poor messaging for filtered-out content may reduce user trust.

## Verification

- Verify Flow projection includes only eligible bonds and participating nodes.
- Verify relayout remains left-to-right and preserves selected atom and viewport focus.
- Verify Flow edges show arrowheads and do not render bond labels.
- Verify Flow nodes use left/right handles only.
- Verify Network view behavior remains unchanged.

## Rollback

- Roll back Flow projection filtering to prior behavior if usability evidence shows severe comprehension loss.
- Keep rollback scoped to Flow-view rendering contracts while preserving shared-dataset assumptions.
