# ADR-260054: Authentication & Authorization Epic Baseline (Adopt Schema 8.1.0)

- **Date**: 2026-06-23
- **Status**: Accepted

## Context

The backend was upgraded to GraphQL schema **8.1.0** (the engine to be renamed `crystord_engine`,
currently vendored at `crystord_api/`). Per `crystord_api/docs/frontend-migration-notes.TEMP.md`, this
is a **breaking** release for the client across five areas: verify-first sign-up, session tokens,
Google linking, taxonomy (Facet→Dimension), and authenticated reads / username-based sharing. The most
urgent consequence is that the old one-shot `signup` is removed, so **new users cannot register at
all**.

The owner walked the full decision register in `AUTH-AUTHZ-EPIC.TEMP.md` and resolved the epic shape.
This ADR records the umbrella decisions; the per-concern decisions are recorded in ADR-260055…260059.
Adoption of 8.1.0 as the pinned contract is approved under `ICR-260001`.

## Decision

1. **Epic scope (D-01):** the *Authentication & Authorization Epic* covers authentication
   (sign-up, sessions, Google, account management, validation, rate limiting) and authorization
   (authenticated reads, error transport, read-side access gating) only.
2. **Boundary (D-22, D-24):** the taxonomy migration (Facet/Category → Dimension/Value) and the
   sharing **write** surface (`shareAtom`/`revokeAtomAccess`/`transferAtomOwnership` + workspaces) are
   **out of scope** and tracked as separate epics. Only the read-side `accessLevel` affordance gating
   (ADR-260059) rides along here because it is pure authorization.
3. **Release shape (D-02):** deliver in **three slices**, S1 first:
   - **S1 (unblocker):** contract adoption, central error mapper, session-token model + expiry
     rewrite, verify-first sign-up, password reset, schema-range bump.
   - **S2 (account management):** Google linking, set/unlink method, email change,
     sign-out-everywhere, account-settings surface, delete account.
   - **S3 (read-side authorization):** `accessLevel` affordance gating.
4. **Schema pin (D-25):** set `public/config.json` `backendSchemaRange` from `^6.0.0` to **`~8.1.0`**
   (auto-adopt `8.1.x` patches; `8.2.0`+ requires a new ICR). Keep the startup `schemaInfo` handshake
   (ADR-260013) as the fail-fast gate.

## Alternatives Considered

- **Single big-bang cutover** of all five breaking areas at once: rejected — larger, slower to restore
  sign-up, and violates the small/reversible-slice delivery policy.
- **Looser `^8.1.0` pin:** rejected — a future 8.x minor could ship unvalidated behavior; `~8.1.0`
  matches the "pin to approved version" governance.

## Consequences

- **Positive**: restores sign-up fastest; keeps each slice reviewable; preserves a clean boundary to
  the taxonomy/sharing epics.
- **Negative**: spans multiple modules (`api-contract`, `auth-entry`, `ui-shell`, graph/details).
- **Risks**: hard backend cut (no dual-version window); rollback viable only while a compatible
  backend is reachable.

## Verification

- `schemaInfo.schemaVersion` satisfies `~8.1.0` at startup; epic E2E flows green per slice; per-slice
  quality gates (lint, typecheck, unit, e2e) pass.

## Rollback

- Redeploy last-known-good artifact pinned to the prior backend and revert `backendSchemaRange`.
