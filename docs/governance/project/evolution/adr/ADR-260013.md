# ADR-260013: GraphQL Schema Compatibility Handshake and Deprecation Governance

## Workflow

- This ADR defines the authoritative B3 API compatibility model for frontend-backend interoperability.
- This ADR supersedes prior generic B3 wording in earlier ADRs and requirements.

Status: Accepted
Date: 2026-04-13
Owners: Repository owner, backend lead, frontend lead

## Context

The backend uses a GraphQL-first architecture with backward-compatible evolution by default and explicit approval required for breaking changes.

The project needs a deterministic compatibility handshake and clear deprecation signaling before release integration.

## Decision

- Schema version storage:
  - Backend exposes a GraphQL-native metadata contract via `Query.schemaInfo`.
  - `schemaInfo` fields:
    - `schemaVersion`: semver string, authoritative for compatibility.
    - `schemaHash`: diagnostic hash of schema.
    - `releasedAt`: RFC3339 timestamp.
- Frontend access:
  - Frontend queries `schemaInfo` once at startup before normal API usage.
  - Frontend validates `schemaVersion` against a configured supported semver range (for example `^1.0.0`).
  - On incompatibility, frontend fails fast with an explicit compatibility error.
  - GraphQL introspection is diagnostic-only and not the primary compatibility source.
- Deprecation communication:
  - Dual-channel signaling is required.
  - In schema: GraphQL `@deprecated` on fields/arguments planned for removal.
  - In governance artifacts: migration paths and sunset timelines in release notes, ICR artifacts, and dedicated deprecation log.

## Alternatives Considered

- Option A: Package-version-only compatibility without GraphQL metadata contract.
- Option B: Separate REST metadata endpoint for schema compatibility.
- Option C: GraphQL-native `schemaInfo` compatibility contract with governance dual-channel deprecation.

## Consequences

- Positive:
  - Compatibility checks become deterministic and machine-readable.
  - Frontend integration failures are caught early during startup.
  - Deprecation governance is explicit and auditable.
- Negative:
  - Requires backend implementation effort for `schemaInfo` and governance discipline.
- Risks:
  - Missing semver range maintenance in frontend config can create false incompatibility failures.

## Verification

- Contract tests validate `schemaInfo` structure and field formats.
- Startup tests validate semver range check and fail-fast behavior.
- Schema checks validate `@deprecated` usage for planned removals.
- Governance checks validate release-notes, ICR references, and deprecation-log entries.

## Rollback

- If rollout issues occur, widen frontend supported semver range under controlled release approval.
- If contract model is replaced, supersede this ADR and migrate references in requirements and backlog.
