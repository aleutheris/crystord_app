# ADR-260046: Explicit Atom Creation via Modal Sidebar

- **Status**: Accepted
- **Date**: 2026-06-05
- **Deciders**: Copilot

## Context and Problem Statement

The existing mechanism for creating a new atom is by double-clicking on an empty area of the canvas. This implicit interaction has several drawbacks:
1.  **Discoverability**: It is not obvious to new users.
2.  **Accidental Creation**: Users may create atoms by mistake while navigating the canvas.
3.  **Inconsistency**: It uses a hardcoded `createAtom('New Atom', ['Node'])` call, which is inconsistent with the rich data that can be provided when updating an atom in the `DetailPanel`.
4.  **Workflow Disruption**: The newly created atom appears on the canvas immediately, regardless of the active search query, which violates the principle that the canvas should always reflect the current search results.

We need a more explicit, consistent, and predictable workflow for atom creation that aligns with the established search-centric interaction model.

## Decision Drivers

- **DD1**: Improve user experience by providing a clear and explicit UI for atom creation.
- **DD2**: Ensure consistency between the fields available for creating a new atom and editing an existing one.
- **DD3**: Prevent accidental atom creation.
- **DD4**: Uphold the integrity of the workspace by ensuring the canvas always displays the results of the active search query.
- **DD5**: Provide clear feedback to the user on the outcome of the creation operation, especially if the new atom is not visible post-creation.

## Considered Options

1.  **Dedicated Pop-up Dialog**: A new modal dialog would appear for atom creation. This is explicit but requires creating and maintaining a new form component, risking divergence from the `DetailPanel`.
2.  **Inline Creation on Canvas**: An editable node appears directly on the canvas. This is visually appealing but complex to manage and can clutter the UI.
3.  **Modal Sidebar**: Reuse the existing `DetailPanel` component, but in a special "creation mode" that overlays the canvas, making the background non-interactive.

## Decision Outcome

Chosen option: **"Modal Sidebar"**. This approach provides the best balance of consistency, user experience, and implementation efficiency.

### Positive Consequences

- **Component Reuse**: We leverage the existing, validated `DetailPanel`, guaranteeing that the fields for creation and editing are always synchronized.
- **Clear, Focused Workflow**: The modal behavior (graying out the background) directs the user's full attention to the creation task.
- **Explicit Action**: A dedicated "Create Atom" button makes the feature highly discoverable.
- **Search Integrity**: The post-creation workflow reinforces the search-first paradigm of the application.

### Negative Consequences

- The `WorkspaceShell` and `DetailPanel` components will require refactoring to manage the new "creation mode" state, but the changes are localized and manageable.

## Implementation Plan

1.  **State Management**: Introduce `isCreatingAtom: boolean` state in `WorkspaceShell`.
2.  **UI Changes**:
    - Add a floating "Create Atom" button to the canvas controls (e.g., in `GraphCanvas` and `NetworkCanvas`).
    - When `isCreatingAtom` is true, render a backdrop overlay to disable interaction with the canvas and other UI elements.
    - The `DetailPanel` will be rendered when `isCreatingAtom` is true, receiving a default/empty atom object and a special `isCreationMode` prop.
3.  **Component Refactoring**:
    - Modify `DetailPanel` to accept an `onCreate` callback and adjust its title/button text when in creation mode.
    - The `onSave` handler will call `onCreate` instead of `onUpdate`.
4.  **Workflow Logic**:
    - The `onCreate` function (passed from `WorkspaceShell`) will call the `graphData.createAtom` mutation.
    - Upon successful creation, it will set `isCreatingAtom` to `false`.
    - It will then trigger a refetch of the current search query via `graphData.search()`.
    - A notification will be displayed confirming the atom was created.
5.  **Cleanup**: The `onDoubleClick` handler on the React Flow canvas will be removed to disable the old creation method.
