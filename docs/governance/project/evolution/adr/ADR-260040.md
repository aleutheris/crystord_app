# ADR-260040: Flow View Projection Mode Toggle and Non-Flow Atom Visibility Refinement

## Workflow

- This ADR records a Flow-view refinement to support authoring workflows that require non-flow atom visibility.
- It supersedes the strict node-exclusion projection rule from ADR-260039 while preserving directional and visual-grammar baselines.

Status: Accepted
Date: 2026-05-19
Owners: Repository owner, product collaborator
Supersedes: ADR-260039 (D1 only)

## Context

ADR-260039 established a strict projection where only eligible bonds and participating nodes are visible in Flow view. That improved readability but created a usability gap: users could not initiate dependency creation from atoms currently outside the strict projected set.

The product now requires a mode-based approach that keeps focused readability by default while enabling optional inclusion for authoring.

## Decision

### D1: Projection Mode Toggle
- Flow view introduces a binary projection mode toggle:
  - Focused mode (default): eligible-bond projection behavior.
  - Include mode: includes non-flow atoms from current search dataset.

### D2: Default and Scope Semantics
- Default mode remains Focused mode to preserve readability-first behavior.
- Include mode is an opt-in authoring aid and does not alter Network view behavior.

### D3: Direction and Relayout Continuity
- Left-to-right relayout contract remains mandatory in both modes.
- Relayout controls and selection/viewport preservation guarantees remain unchanged.

### D4: Visual Differentiation
- Non-flow atoms in Include mode must be visually distinguishable from flow-participating atoms.
- Existing Flow edge grammar remains unchanged for this refinement:
  - directed arrowheads remain visible,
  - bond labels remain not rendered in this batch.

### D5: Scope Boundary
- This ADR does not introduce top-to-bottom orientation toggles.
- Orientation toggle remains deferred future scope.

## Alternatives Considered

- Option A: Keep strict-only projection with no inclusion mode. Rejected: harms dependency-authoring usability.
- Option B: Always include all atoms in Flow view. Rejected: reduces readability and weakens flow semantics.
- Option C: Force users to switch to Network view for authoring. Rejected: unnecessary context switching for a common task.

## Consequences

### Positive
- Preserves readability-first default while enabling practical authoring workflows.
- Reduces need to context-switch to Network view for dependency creation.

### Negative
- Adds a new control surface and corresponding state/test complexity.
- Include mode may increase visual density in larger result sets.

### Risks
- Weak visual differentiation in Include mode can confuse users.
- Toggle state handling could create inconsistent user expectations if not explicit.

## Verification

- Verify default mode is Focused mode.
- Verify Include mode displays non-flow atoms.
- Verify left-to-right relayout remains enforced in both modes.
- Verify visual distinction of non-flow atoms in Include mode.
- Verify Flow edge grammar and Network view behavior remain unchanged.

## Rollback

- If inclusion mode causes severe readability regressions, disable Include mode and revert to strict focused projection while retaining other ADR-260039 baselines.
