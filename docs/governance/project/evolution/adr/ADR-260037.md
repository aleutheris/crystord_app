# ADR-260037: Network View Connection Preview and Ring Token Semantics Hardening

## Workflow

- This ADR records a focused refinement following BI-260037 delivery feedback.
- It separates non-compliance correction from scope evolution.
- It formalizes new behavior for connection preview rendering and ring color token semantics.

Status: Accepted
Date: 2026-05-14
Owners: Repository owner, product collaborator
Supersedes: -

## Context

Post-delivery validation identified two categories:

1. Non-compliance against accepted behavior: connection completion should accept drop over the target circle (not discrete handle points only).
2. New expectations not yet formalized in requirements: straight-line drag preview and explicit visual distinction between outer initiation ring and inner selection visual, implemented through centralized color tokens.

The governance model should preserve existing requirements for non-compliance fixes while introducing new requirements for newly requested behavior.

## Decision

### D1: Non-Compliance Is Corrective Delivery, Not Requirement Rewrite
- Existing accepted requirement language for target-circle drop remains unchanged.
- Corrective implementation and stronger tests are tracked through a new backlog item.

### D2: Straight-Line Drag Preview Is a New Functional Baseline
- During active Network-view connection drag, preview rendering uses a straight line.
- Established-edge anchoring and arrowhead semantics remain governed by existing requirements.

### D3: Outer Ring vs Inner Selection Visual Must Be Distinct and Tokenized
- Outer connection-initiation ring and inner selection visual use distinct colors.
- Colors must come from centralized tokens, not local hardcoded literals in feature components.

### D4: Test Hardening Is Mandatory for Both Categories
- Corrective tests must validate target-circle drop completion across varied release points.
- New-behavior tests must validate straight preview rendering and tokenized color sourcing.

## Alternatives Considered

- Option A: Reopen and edit existing requirements for non-compliance. Rejected: weakens traceability and blurs bug-fix vs scope-change.
- Option B: Implement new behavior without requirements. Rejected: creates governance drift.
- Option C: Keep hardcoded colors temporarily. Rejected: conflicts with branding-governance discipline.

## Consequences

### Positive
- Governance remains explicit: non-compliance and scope evolution are handled with different records.
- Future branding changes are easier due to token centralization.
- Regression risk is reduced through targeted interaction tests.

### Negative
- Adds additional governance and test scope before implementation can be considered complete.

### Risks
- If token boundaries are underspecified, color usage may fragment again.
- If E2E coverage is too narrow, circle-hit completion regressions may reappear.

## Verification

- Traceability check: new REQ/BI/ADR links are complete and indexed.
- Corrective behavior check: release over target circle completes bond creation.
- Preview behavior check: drag preview edge is straight.
- Token governance check: interaction colors are sourced from centralized tokens.

## Rollback

- Do not rollback corrected target-circle drop behavior.
- If visual-token changes cause unacceptable regressions, temporarily pin centralized token values while preserving token indirection.
