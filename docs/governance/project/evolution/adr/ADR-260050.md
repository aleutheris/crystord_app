# ADR-260050: Consolidated Authentication Strategy

- **Date**: 2026-06-08
- **Status**: Accepted

## Context

The application requires a robust and user-friendly authentication system that provides multiple entry points for different user needs. Previous implementations addressed credential, Google, and demo access in separate backlog items (`BI-260011`, `BI-260048`, `BI-260049`). To ensure a cohesive user experience and a maintainable architecture, a consolidated strategy is needed.

This ADR formalizes the decision to implement a single, unified authentication component that supports three distinct methods:
1.  **Credential-Based Authentication**: Standard sign-up and sign-in with a username and password.
2.  **Google Sign-In**: Passwordless authentication using a Google account.
3.  **Demo Mode**: One-click access using a pre-configured, non-persistent demo account.

## Decision

We will implement a unified `SignInPage` component that serves as the central hub for all authentication flows.

1.  **Component Structure**: The `SignInPage` will present the user with clear choices for each authentication method. It will include a form for username/password, a prominent "Sign in with Google" button, and a "Try a Demo" button.
2.  **Credential Flow**:
    - Users can sign up for a new account.
    - Registered users can sign in using their credentials.
    - The `AuthProvider` will handle the logic for communicating with the backend to validate credentials and manage session tokens.
3.  **Google Sign-In Flow**:
    - The implementation will leverage the Google Identity Services (GIS) library.
    - Upon successful authentication with Google, the received `idToken` will be sent to the backend to verify the user's identity and issue a session token. No separate sign-up step is required for Google users.
4.  **Demo Mode Flow**:
    - A "Try a Demo" button will trigger a `signInAsDemoUser` function.
    - This function will authenticate with the backend using hardcoded demo credentials (`demo`/`demo`).
    - The session will be flagged as a demo session, and no user data will be persisted.
5.  **State Management**: A central `AuthProvider` will manage the user's authentication state, including the session token and whether the user is in demo mode. This provider will make the state available throughout the application via a React Context.

## Consequences

- **Positive**:
    - Provides a clear, consistent, and user-friendly authentication experience.
    - Simplifies the architecture by centralizing authentication logic.
    - Reduces code duplication and improves maintainability.
    - Aligns with modern best practices for web application authentication.
- **Negative**:
    - Requires refactoring of any existing, fragmented authentication components to use the new unified `SignInPage` and `AuthProvider`.
- **Neutral**:
    - This decision supersedes the architecture and implementation details of `ADR-260014`, `ADR-260048`, and `ADR-260049`.
