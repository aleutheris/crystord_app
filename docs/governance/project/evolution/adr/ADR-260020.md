# ADR-260020: Delivery and Quality Baseline for MVP

## Workflow

- This ADR records accepted I1-I6 delivery and quality decisions.
- The decision is accepted because release confidence and user trust depend on explicit quality gates and operational safeguards.

Status: Accepted
Date: 2026-04-17
Owners: Repository owner, product collaborator

## Context

The project has established product, UX, architecture, and stack choices.

A delivery-and-quality baseline is now required so MVP release decisions are based on objective flow validation, reliability expectations, and rollback readiness.

## Decision

- I1 accepted decision:
  - MVP release gates include end-to-end validation of six critical flows: entry, bootstrap search load, atom create/edit, bond create/name, delete safety behaviors, and error-path resilience.
- I2 accepted decision:
  - Accessibility baseline is WCAG 2.1 AA practical coverage for core workflows (keyboard access, focus visibility, semantic labels, contrast).
- I3 accepted decision:
  - Observability baseline includes structured client logs, contextual error events, and key workflow metrics (workspace load, search latency, mutation outcomes, compatibility/auth failures).
- I4 accepted decision:
  - Unacceptable failures include unrecoverable crashes, silent mutation data loss, route-guard failures, frozen graph interactions, and non-actionable blocking errors.
- I5 accepted decision:
  - Release and rollback model requires preflight gates, versioned artifacts, immediate post-release smoke checks, and rollback via last-known-good redeploy.
- I6 accepted decision:
  - Placeholder fallback behavior must be explicit, honest, and non-blocking, with clear route back to workspace.

## Alternatives Considered

- Option A: Lightweight release checklist with qualitative quality judgment.
- Option B: Explicit quality-gated release model with measurable baselines.
- Option C: Delay quality baseline until after first MVP release.

## Consequences

- Positive:
  - Improves release confidence through explicit, testable gates.
  - Reduces risk of trust-damaging placeholder and error behaviors.
  - Clarifies rollback response when incidents occur.
- Negative:
  - Adds process overhead before release.
  - Requires disciplined telemetry and test maintenance.
- Risks:
  - Incomplete observability instrumentation could weaken incident diagnosis.

## Verification

- CI/release checks confirm I1 critical flow validations pass before deployment.
- Accessibility checks confirm baseline coverage on primary interaction routes.
- Observability checks confirm required logs/events/metrics are emitted.
- Operational drills confirm rollback procedure can restore last-known-good build.

## Rollback

- Supersede this ADR if delivery evidence shows baseline is too strict or insufficient.
- Update linked requirements and backlog artifacts to the superseding decision.
