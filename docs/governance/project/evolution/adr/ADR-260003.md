# ADR-260003: Secondary Users and Backend Contract Definition

## Workflow

- This ADR records accepted decisions about secondary personas and the GraphQL backend contract that grounds frontend implementation.
- The decision is accepted because it determines integration scope, schema stability strategy, and error-handling requirements.

Status: Accepted
Date: 2026-04-13
Owners: Repository owner, product collaborator

## Context

Secondary personas and backend-contract clarity are required so post-MVP extensibility and developer experience remain aligned with the core product.

A stable, versioned contract ensures frontend changes do not break unexpectedly and integration partners can plan their work.

## Decision

- A3 accepted decision:
  - Secondary users in post-MVP are developers and integrators extending the system, and no-code/spreadsheet users adopting Crystord as a connected-data companion.
- B1 accepted decision:
  - Backend contract source is the GraphQL schema as documented in the backend user guide plus GraphQL introspection API.
- B2 accepted decision:
  - Atoms, Bonds, and Nuclearies are editable in MVP; Shellies, Operations, Constants, and label administration are read-only or deferred.
- B3 accepted decision:
  - Superseded by ADR-260013, which defines the `Query.schemaInfo` contract, startup semver validation, and dual-channel deprecation policy.
- B4 accepted decision:
  - Essential operations are retrieve, change, destroy, and signin (with fixed demo credentials in development/testing); list_labels, signup, and Google sign-in are deferred to post-MVP.
- B5 accepted decision:
  - Frontend must handle network/timeout errors, GraphQL validation errors, auth errors, rate limits, and domain errors; deprecation warnings are logged.

## Alternatives Considered

- Option A: Defer all backend-contract decisions to implementation time.
- Option B: Accept secondary personas and contract only for MVP, then decide per-feature post-MVP.
- Option C: Accept both secondary personas and a stable contract strategy now.

## Consequences

- Positive:
  - Schema stability enables team confidence in integration work.
  - Secondary-user personas guide post-MVP direction without distorting MVP scope.
- Negative:
  - Contract versioning requires discipline in backend release management.
- Risks:
  - Label administration deferral could become friction if users expect label creation in MVP.

## Verification

- Contract smoke tests validate essential mutations and queries work with MVP atoms/bonds.
- Label query and atom retrieval tests confirm read-only label behavior.
- Error case tests confirm network and GraphQL-error handling.

## Rollback

- Supersede this ADR if secondary-persona priorities or backend-schema strategy change after MVP validation.
- Update linked requirements and backlog items.
