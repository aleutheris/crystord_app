# ADR-260017: Information Architecture Baseline for MVP Workspace Application

## Workflow

- This ADR records accepted F1-F6 information-architecture decisions.
- The decision is accepted because route boundaries, module seams, and ownership rules directly determine maintainability and delivery speed.

Status: Accepted
Date: 2026-04-17
Owners: Repository owner, product collaborator

## Context

The project already commits to a graph-first workspace, in-workspace navigation, side-panel editing, and demo sign-in entry.

Information architecture must now define top-level areas, module boundaries, extension seams, content ownership, and public-vs-authenticated shell separation so implementation stays coherent.

## Decision

- F1 accepted decision:
  - MVP top-level areas are Entry (demo sign-in), Workspace (graph/search/editing), and minimal supporting placeholder pages.
- F2 accepted decision:
  - Administration remains Post-MVP (label governance, user/role control, configuration, audit/release operations).
  - MVP may expose placeholder-only admin routes.
- F3 accepted decision:
  - Feature-oriented bounded areas are: auth-entry, workspace-graph, workspace-search, workspace-details, api-contract, ui-shell.
- F4 accepted decision:
  - Define extension seams without plugin runtime: search-filter registry, graph renderer decorators, side-panel section slots, action/command registry, placeholder route slots.
- F5 accepted decision:
  - Ownership hierarchy: workspace context owns visible graph state; atoms are primary entities; bonds represent relationships; nuclearies belong to atoms; labels are referenced metadata only.
- F6 accepted decision:
  - Soft split in one frontend app via route/layout boundaries: public shell for sign-in/placeholders, authenticated shell for workspace flows.

## Alternatives Considered

- Option A: Monolithic route/component structure with minimal boundaries.
- Option B: Feature-bounded architecture with explicit shell and domain seams.
- Option C: Multiple separately deployed frontends from MVP start.

## Consequences

- Positive:
  - Improves maintainability through clear domain boundaries.
  - Supports iterative growth through defined extension seams.
  - Keeps deployment simpler with a single application runtime.
- Negative:
  - Requires disciplined module-interface governance from early implementation.
  - Placeholder routes need clear UX handling to avoid confusion.
- Risks:
  - Boundary drift can occur if feature code bypasses shared contracts.

## Verification

- Architecture review confirms feature modules and dependency boundaries are implemented.
- Route checks confirm public/authenticated shell split and guard behavior.
- Scope checks confirm admin is placeholder-only in MVP and full admin behaviors are excluded.
- Data-flow checks confirm content ownership hierarchy is preserved across graph/search/detail workflows.

## Rollback

- Supersede this ADR if architecture evidence shows the current split is insufficient.
- Update linked requirements and backlog entries to the superseding decision.
