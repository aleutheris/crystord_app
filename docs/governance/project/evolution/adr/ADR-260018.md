# ADR-260018: Frontend Architecture Baseline for MVP

## Workflow

- This ADR records accepted G1-G5 frontend architecture decisions.
- The decision is accepted because architecture, state partitioning, and contract rules determine implementation quality and change safety.

Status: Accepted
Date: 2026-04-17
Owners: Repository owner, product collaborator

## Context

Prior decisions established graph-first interaction, in-workspace navigation, side-panel editing, search bootstrap, and information-architecture boundaries.

The project now needs an explicit frontend architecture baseline covering architectural style, state partitioning, rendering model, modular dependency rules, and contract-stability controls.

## Decision

- G1 accepted decision:
  - Use feature-sliced modular architecture organized by capability/domain boundaries.
- G2 accepted decision:
  - Partition state into server state, shared workspace state, feature-local state, and transient UI state with explicit adapter seams.
- G3 accepted decision:
  - Use CSR-first rendering for the interactive workspace to optimize runtime interaction latency.
- G4 accepted decision:
  - Enforce module boundaries through public APIs and dependency rules; features do not import each other's internals and GraphQL bindings are centralized in api-contract.
- G5 accepted decision:
  - Protect internal contracts via explicit public interfaces, contract tests for cross-feature payloads/adapters, and approval gating for breaking changes.

## Alternatives Considered

- Option A: Monolithic layer-first structure with broad shared dependencies.
- Option B: Feature-sliced modular architecture with strict boundaries.
- Option C: Micro-frontend split from MVP start.

## Consequences

- Positive:
  - Improves maintainability by reducing cross-feature coupling.
  - Makes integration behavior testable through explicit contracts.
  - Keeps runtime behavior optimized for interactive graph workflows.
- Negative:
  - Requires upfront discipline for boundary and contract governance.
  - May add modest initial setup overhead for contract and boundary checks.
- Risks:
  - Boundary erosion if dependency rules are not continuously enforced.

## Verification

- Architecture checks confirm feature-sliced modules and approved dependency directions.
- State checks confirm the four-state-category partition and adapter boundaries.
- Runtime checks confirm CSR-first workspace behavior and interaction responsiveness.
- Contract checks confirm cross-feature payload tests and breaking-change gate enforcement.

## Rollback

- Supersede this ADR if implementation evidence shows the architecture is insufficient.
- Update linked requirements and backlog items to the superseding decision.
