# ADR-260052: Platform Token Handoff via URL Parameter

- **Date**: 2026-06-10
- **Status**: Accepted

## Context

The Crystord platform (`crystord.com`) and the Crystord workspace app (`app.crystord.com`) are served from different origins. Because `localStorage` is partitioned by origin, a token obtained during sign-in on the platform page cannot be read by the app page directly.

When the platform redirects a user to the workspace app, the app must receive the authenticated token so that the user does not have to sign in again. A shared-cookie or server-side session approach would require backend infrastructure changes. A URL query parameter handoff is sufficient for this use case: the token is short-lived, the parameter is stripped from the URL immediately after consumption before any user interaction, and no third-party scripts are loaded on the app page that could intercept the referrer.

The accepted tradeoff is that the token will briefly appear in browser history and server access logs. Given the token's expiry and the internal nature of the tool, this is acceptable.

## Decision

When the workspace app loads, it checks for a `token` query parameter in the URL. If present:

1. The token is consumed by calling `signIn(token)`, which persists it to `localStorage` exactly as a normal sign-in would.
2. The `token` parameter is immediately stripped from the URL using `history.replaceState`, so it does not remain in browser history beyond the initial load.
3. The handoff takes effect regardless of whether a previous session token exists in `localStorage`; the URL token is always authoritative on redirect.

The logic is encapsulated in a dedicated `useUrlTokenHandoff` hook, called from `AuthProvider` on mount.

## Consequences

- **Positive**:
    - Users redirected from the platform are signed in automatically without a second login prompt.
    - No backend changes required.
    - Implementation is small and isolated to one hook.
    - The token is removed from the URL before the user can copy or share it.
- **Negative**:
    - The token appears momentarily in the browser history entry for the initial load URL and in server access logs.
- **Neutral**:
    - The platform page is responsible for appending `?token=<jwt>` to the redirect URL; this app consumes it passively.
