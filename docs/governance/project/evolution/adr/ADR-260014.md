# ADR-260014: User Experience Model for Graph Workspace MVP

## Workflow

- This ADR records accepted C1-C7 user-experience decisions for MVP delivery.
- The decision is accepted because interaction and entry-flow choices directly shape architecture, usability, and delivery risk.

Status: Accepted
Date: 2026-04-17
Owners: Repository owner, product collaborator

## Context

The product direction is graph-first, but graph interaction is not useful if users enter an empty workspace with no discoverable dataset context.

The UX model must define how users enter the application, preload data context, navigate in MVP, and avoid non-core page complexity.

## Decision

- C1 accepted decision:
  - Interaction style is graph-first with search bootstrap.
  - On workspace load, the frontend runs a default search to preload an initial atom/bond set.
  - Users can immediately refine scope with label search controls.
- C2 accepted decision:
  - Workspace model is a full workspace centered on the graph canvas.
- C3 accepted decision:
  - Navigation model is in-workspace tabs/panels instead of route-heavy context switching.
- C4 accepted decision:
  - Editing model is side-panel driven from graph selection.
- C5 accepted decision:
  - Responsiveness target is desktop-only for MVP.
- C6 accepted decision:
  - MVP entry requires demo sign-in in development/testing with fixed credentials (`demo` / `demo`) before workspace access.
  - Production-grade multi-user auth remains out of MVP scope.
- C7 accepted decision:
  - Non-core pages may ship as minimal placeholders with baseline branding/navigation and reliable return to workspace.

## Alternatives Considered

- Option A: Search-first or form-first interaction as the default model.
- Option B: Graph-first interaction with bootstrap search and side-panel editing.
- Option C: Full production auth in MVP.

## Consequences

- Positive:
  - Preserves graph-first product identity while avoiding empty-canvas dead starts.
  - Reduces cognitive and navigation overhead for primary users.
  - Validates a real entry/auth step in MVP without full auth complexity.
- Negative:
  - Bootstrap search defaults may bias first-view context if not tuned carefully.
  - Desktop-only target excludes tablet/mobile workflows in MVP.
- Risks:
  - Demo credentials could leak into production if environment controls are weak.

## Verification

- Validate startup behavior runs default search and renders non-empty graph context when data exists.
- Validate users can refine context using label search without leaving workspace.
- Validate successful demo sign-in is required before workspace entry in development/testing.
- Validate non-core placeholder pages retain branding/navigation and allow return to workspace.

## Rollback

- Supersede this ADR if user testing shows bootstrap search or side-panel editing is ineffective.
- Update linked requirements and backlog items to the superseding decision.
