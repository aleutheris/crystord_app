# ADR-260035: Force-Directed Automatic Layout for Network View

## Workflow

- This ADR records the accepted automatic layout strategy for Network view.
- It replaces the fixed square-grid initial placement with topology-aware force-directed layout.
- This ADR complements ADR-260033 (anchoring) and ADR-260034 (direction arrows) as the third refinement in the Network-view rendering improvement sequence.

Status: Accepted
Date: 2026-05-13
Owners: Repository owner, product collaborator

## Context

Network view currently places atoms in a fixed square grid using `Math.sqrt(atoms.length)` columns with no awareness of bond topology. This means:
- Strongly connected atoms may be placed far apart.
- Atoms with no bonds sit adjacent to densely-connected ones.
- The visual arrangement gives no structural signal beyond node/edge existence.

A topology-aware layout algorithm would place connected atoms closer together and spread unconnected atoms out, making graph structure immediately readable without manual rearrangement.

Bond graphs in this domain are not DAGs — crystals form cyclic, undirected-topology bond structures — so hierarchical algorithms (Dagre, ELK) are inappropriate.

## Decision

### D1: Algorithm — Force-Directed via d3-force
- Use `d3-force` as the layout simulation engine.
- Apply repulsion between all nodes (`forceManyBody`) and attraction along edges (`forceLink`).
- `d3-force` is the established standard for undirected graph layout and is already the demonstrated integration pattern in @xyflow/react documentation. It is small, focused, and tree-shakeable.
- No new heavyweight dependency is introduced.

### D2: Layout Trigger — Initial Load Only, Preserve Drag Positions
- Layout is computed once: when the first search result set is rendered and no stored node positions exist.
- The existing `mergeNodePositions` pattern is reused: atoms with stored drag positions are never repositioned by the auto-layout run.
- A "Re-layout" toolbar action allows users to discard stored positions and recompute layout on demand.

### D3: Synchronous Iteration Cap — No Mid-Paint Animation
- The force simulation runs synchronously and is stopped after a fixed iteration cap (~300 ticks) before the first paint.
- Animated transitions between positions are not used for initial layout; the graph appears fully arranged.
- This avoids distracting mid-render movement and keeps implementation straightforward.

### D4: Degrade Thresholds — Respect Existing Policy
- ≤ 150 nodes: full layout computation.
- > 150 and ≤ 400 nodes: layout computation proceeds in reduced mode (no label data propagated).
- > 400 nodes: layout computation is skipped; grid fallback is used. This is consistent with REQ-QR-260003 and BI-260031 blocked-render policy.

### D5: Minimum Separation Constraint
- A minimum separation of `NODE_SIZE + padding` is enforced to prevent nodes from overlapping after simulation.
- Self-loop edges are excluded from `forceLink` inputs (a node cannot attract itself).

### D6: Non-Determinism is Acceptable
- Force-directed layout is non-deterministic across runs. The same graph may produce different arrangements on different loads.
- This is acceptable for an exploratory workspace. Users who want a stable arrangement may save it via manual drag (D2 preserves it).
- Seeded PRNG for reproducibility is deferred to a future requirement if stability becomes a user need.

## Alternatives Considered

- Option A: Retain fixed square grid. Rejected: gives no topological signal; visually misleading for graph exploration.
- Option B: Hierarchical layout (Dagre/ELK). Rejected: assumes DAG structure; crystal bond graphs are cyclic and undirected-topology; would produce degenerate or incorrect layouts.
- Option C: Animated layout on every data change. Rejected: distracting in a workspace tool; adds implementation complexity for minimal benefit.
- Option D: Run layout async/incrementally with animation. Rejected: mid-paint jank is worse than a synchronous pre-render cap.

## Consequences

### Positive
- Connected atoms cluster naturally, making graph topology immediately readable.
- No new heavyweight library dependency.
- Consistent with @xyflow/react integration patterns.
- Drag-position preservation keeps user intent stable across data refreshes.

### Negative
- Layout is non-deterministic; same graph may appear differently on reload.
- Synchronous simulation on large graphs (approaching 400 nodes) may add measurable initial-render latency.

### Risks
- Iteration cap tuning: too few ticks → nodes cluster poorly; too many → noticeable initial latency. Tuning required at implementation time.
- Self-loop exclusion must be correctly implemented or d3-force will produce NaN positions.
- Merging logic must correctly distinguish "no prior position" (layout) from "has prior position" (preserve); regression risk if `mergeNodePositions` semantics change.

## Verification

- Unit tests validate that `atomsToNetworkNodes` (or equivalent) produces topology-influenced positions (connected atoms nearer than unconnected ones for a test fixture).
- Unit tests validate that `mergeNodePositions` preserves existing positions and only fills in missing ones.
- Unit tests validate that self-loop edges are excluded from layout input.
- Unit tests validate that node count above 400 falls back to grid layout without simulation.
- Integration tests validate re-layout button discards stored positions and recomputes.
- E2E tests validate that initial Network view renders without visible mid-paint layout animation.

## Rollback

- If force layout causes render latency complaints, reduce iteration cap first.
- If non-determinism becomes a usability issue, add seeded PRNG with a stable seed derived from the atom set.
- If d3-force introduces bundle-size concerns, evaluate @xyflow/react's own layout utilities as a drop-in replacement.
