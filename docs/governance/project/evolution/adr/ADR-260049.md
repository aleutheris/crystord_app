# ADR-260049: Add Username/Password Authentication

- **Date**: 2026-06-08
- **Status**: Accepted

## Context

The application currently supports authentication only via Google Sign-In. To broaden accessibility and cater to users who prefer not to use third-party identity providers or do not have a Google account, a direct username/password authentication method is required. This is also necessary to support non-human users or specific internal roles (like a 'demo' user) that cannot use Google Sign-In.

## Decision

We will implement a standard username/password authentication flow. This includes:
1.  A sign-in form that accepts a username/email and a password.
2.  Client-side logic to handle user input and communicate with the backend authentication endpoint.
3.  A temporary compatibility layer where the input from the username/email field is passed as the `email` parameter to the backend, until the backend service is updated to handle both.

This approach allows for immediate frontend development and support for demo/password-based accounts, while providing a clear path for full integration once the backend is ready.

## Consequences

- **Positive**:
  - Increased accessibility for users without a Google account.
  - Enables authentication for system-level or demo accounts.
  - Aligns with standard application authentication patterns.
- **Negative**:
  - Introduces the need to manage user credentials securely.
  - Adds complexity to the authentication logic.
  - The initial implementation has a temporary shim to accommodate the backend, which will need to be removed later.
