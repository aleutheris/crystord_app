# ADR-260034: Directional Arrowheads for Network-View Bonds

## Workflow

- This ADR records the accepted direction-visibility baseline for Network-view bonds.
- It adds explicit directional arrowheads while preserving label optionality and existing data contracts.
- This ADR complements ADR-260033 by improving edge semantics after closest-point anchoring refinement.

Status: Accepted
Date: 2026-05-13
Owners: Repository owner, product collaborator

## Context

Network view aims for readable graph semantics under growing density. Closest-point anchoring improves edge geometry, but users still need immediate direction cues for bond interpretation.

Label-only direction signaling creates clutter and weak scanability in dense graphs. A compact, non-color directional marker is needed.

## Decision

### D1: Arrowhead Baseline
- All directed Network-view bonds render a target-end arrowhead.
- Arrowhead orientation follows source-to-target bond direction.

### D2: Reduced-Mode Behavior
- Arrowheads remain visible in reduced mode.
- Bond text labels may remain omitted by default in reduced mode and full mode to preserve readability baseline.

### D3: Accessibility and Discoverability
- Direction semantics do not rely on color alone.
- Selected-state emphasis can increase stroke/marker prominence without changing direction meaning.

### D4: Scope Boundary and Extensibility
- Applies only to Network view rendering.
- Flow view semantics and backend bond model remain unchanged.
- Edge label data remains preserved as a future seam for optional label enablement.

## Alternatives Considered

- Option A: Keep direction implied without arrowheads. Rejected: weak discoverability and higher interpretation ambiguity.
- Option B: Use labels only for direction. Rejected: increases clutter and scales poorly on dense graphs.
- Option C: Add arrowheads only in full mode. Rejected: reduced mode still needs directional meaning.

## Consequences

### Positive
- Faster directional parsing in dense views.
- Better non-color accessibility cue for edge semantics.
- Preserves compatibility with optional future label rendering.

### Negative
- Slightly increases visual density per edge.
- Requires marker-style consistency across states.

### Risks
- Oversized markers can clutter small subgraphs.
- Multi-edge overlap may require future offset/curvature tuning.

## Verification

- Unit tests validate marker configuration on mapped Network-view edges.
- Integration tests validate selected/unselected and reduced/full rendering marker consistency.
- E2E tests validate arrow direction matches source-to-target relation for loaded and newly created bonds.
- Regression checks validate Flow view behavior remains unchanged.

## Rollback

- If marker density negatively impacts usability, reduce marker size first while preserving directional visibility.
- If necessary, temporarily disable selected-state marker enlargement but keep baseline arrowheads.
